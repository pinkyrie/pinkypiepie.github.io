关于软件开发的问题

software

  • 情报处理system的program… 关联文书

    • 包含要求定义书、外部设计书、内部设计书、database定义书、coding规约、

  • computer装置

    • application software(mailer,browzer,word-processor)
    • middleware(database manager)
    • basic software(operating system)

software vs hardware

  • software–>机能拓张 性能改善

    • 经年变化无
    • 导入后修正可能
    • 量产/分发/销售cost低

  • hardware–>维持机能和性能

    • 经年变化有
    • 导入后修正基本不可能
    • 量产和分发cost高

software 开发

  • software形式 实现顾客的要求
  • balance 品质 and cost

history

  • 情报system化
  • 开发计划 life cycle
  • 开发效率（生产性）重视
  • 社会依赖度++ 易用性要求++
  • 复杂度++

software engineering

  • 构筑 运用 保守
  • 生产性 品质

目的

  • 品质特性++

software crisis

  • computer性能++ computer需求++
  • main problem:
    • 由于软件生产物增大（computer性能向上和普及导致software大规模化和数量增加）
    • ->software相关人员++（skateholder多样化）
    • ->期间长期化（开发时间和利用时间++ 动作环境和社会要求变动）
    • ->对社会影响变化（computer领域的扩大，对社会的的重要性++ 信赖性要求++ software大众化导致易用性++）

大规模 高信赖开发导向

  • 大规模 复数的人开发 很多stakeholder 实际的利用运用（效率性保守性信赖性）
  • 实践：不光是制作还有运用保守的考虑

software品质特性

  • functional suitability（机能适合性）
    • Functional suitability is measured value of comparison between things that software can actually do and things that software should do.（有没有实现必要的功能）
    • 副特性：机能完全性 机能正确性 机能适切性
  • performance efficiency（机能效率性）
    • 使用资源的量来衡量
    • 副特性：时间效率性 资源效率性 容量满足性
  • compatibility（互换性）
    • 兼容性 与其他环境共存容易
    • 举例：同一个系统的不同版本（比如windows会有同一个功能多个冗余函数以兼容不同版本的系统）
    • 区分和移植性
    • 副特性：共存性 相互运用性
  • usability（使用性）
    • 对使用者使用容易
    • 副特性：适当度识别性、可习得性、操作可操作性、用户防错性、用户界面快美性、可访问性
  • reliability（信赖性）
    • 继续正常运行
    • 副特性：成熟性 可用性 障害许容性 回复性
  • security
    • 安全
    • 副特性：机密性 integrity 否认防止性 责任追踪性 真正性
  • maintainability（保守性）
    • 修订作业必要的劳力少
    • 副特性：module性 再利用性 解析性 修正性 试验性
  • portability（移植性）
    • 其他环境移植容易
    • 副特性：适应性 设置性 置换性

解决问题方法

分割统治和构造化（divide and conquer）

  • 大问题 -> 独立性高的小问题
  • 例子：分割软件成为部品 构造化分析/构造化programming

抽象化（abstraction）和modeling

  • 除去非本质部分

  • 抽取出需要讨论项目相关的重要部分进行记述（记述的东西称作model）

  • modelling 为了使本质浮出

  • 庞大复杂的系统可以不拘泥细节 抓住本质

  • 存在复数的model：每个项目都有不同的model 相同项目由不同的观点可以制作不同模型

要求分析

  • 要求（requirements）：软件在什么条件下提供什么服务 需要明快、详细作为文书的严密定义
  • 要求分析：system的service和制约检查得出、分析、文书化过程（执行不充分会导致软件开发的失败）
  • 要求工学（requirements engineering :RE）总结要求、软件应该制作什么

追踪可能性（traceability）

  • 软件开发有许多的判断（fork分支）需要记录判断了什么、使用了什么方法方式选择的

  • 举例：要求仕样和设计仕样 成果物artifact文书化记录（交付件）开发过程有很多artifact

  • traceability：从某一成果物中寻找其他成果物的对应位置和大元判断的性质

  • 经验积蓄和再利用

  • 系统的评价和管理
